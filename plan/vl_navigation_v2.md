# 大众点评 VL 驱动自动化 Plan v2

## 核心变化

**v1 方案（废弃）：** 采集截图 → VL 标注 → YOLO 训练 → 端侧推理
**v2 方案：** 截图 → VL 实时识别节点 → 点击 → 再截图 → 循环

**为什么放弃 YOLO：**
- 大众点评用自绘控件，无障碍服务拿不到内容节点
- VL（Qwen2.5-VL-72B）直接就能识别页面元素和归一化坐标
- 不需要训练 YOLO，VL 本身就是"眼睛"

---

## 架构

```
手机端                    服务器
┌────────────┐           ┌─────────────────────┐
│ 截图        │  base64   │ 1. 保存截图          │
│ 上传        │ ────────→ │ 2. VL 识别节点       │
│             │           │    (归一化坐标 0-1)   │
│             │  JSON     │ 3. 记录页面节点树     │
│ 收到节点列表 │ ←──────── │ 4. 返回可点击节点     │
│ 点击节点    │           │                     │
│ 再截图上传  │ ────────→ │ 循环...              │
└────────────┘           └─────────────────────┘
```

**关键设计：**
- 所有坐标归一化（0-1），乘以 device.width/height 得到实际像素
- 换手机不需要改任何代码
- VL 返回的节点树按页面类型存储，建立"页面地图"
- 采集和执行共用同一套 VL 识别逻辑

## Phase 0：VL 节点识别验证

### 目标
验证 VL 能稳定识别大众点评页面的可点击节点，返回准确的归一化坐标。

### 步骤
1. 手机截图上传 → 服务器调用 VL
2. VL 返回：`[{label, text, x_center, y_center, width, height}]`
3. 手机按归一化坐标点击
4. 验证点击是否准确命中目标

### 验收标准
- VL 识别首页 ≥ 6 个卡片节点
- 点击准确率 ≥ 80%（5 次点击 ≥ 4 次命中正确的卡片）
- 坐标归一化，换分辨率不需要改代码

---

## Phase 1：页面地图采集

### 目标
自动浏览大众点评各页面，用 VL 识别每个页面的节点树，建立页面地图。

### 采集流程
```
首页 → VL识别节点 → 点击某个卡片
  → 笔记详情页 → VL识别节点 → 记录 → back
  → 点击另一个卡片 → 笔记详情页 → 记录 → back
  → 滑动首页 → VL识别新节点 → 继续...
```

### 页面节点树格式
```json
{
  "page_type": "首页",
  "nodes": [
    {
      "label": "内容卡片",
      "text": "这也太好吃了吧",
      "x": 0.25, "y": 0.25,
      "w": 0.45, "h": 0.28,
      "clickable": true
    },
    {
      "label": "搜索栏",
      "x": 0.5, "y": 0.06,
      "w": 0.8, "h": 0.04,
      "clickable": true
    }
  ],
  "navigation": {
    "bottom_tabs": ["首页", "地图", "发布", "消息", "我的"],
    "top_bar": "搜索栏"
  }
}
```

### 存储规则
- 每个页面类型只存一份节点树模板
- 新发现的页面类型追加
- 节点坐标全部归一化

### 先覆盖的页面（不贪多）
1. **首页**（双列瀑布流）
2. **笔记详情页**（点卡片进入的页面）

### 验收标准
- 建立 ≥ 2 个页面的节点树
- 每个页面的核心交互元素都被识别
- 节点树存储在服务器，可查询

## Phase 2：VL 驱动自动操作

### 目标
基于页面地图，实现 VL 驱动的自动操作循环。

### 操作流程
```
截图 → VL识别当前页面类型和节点
     → 匹配页面地图，确定操作策略
     → 执行操作（点击/滑动/输入）
     → 再截图 → 循环
```

### 验收标准
- 能从首页自动进入笔记详情页并返回
- 连续 10 次操作无崩溃
- 操作结果与预期一致率 ≥ 80%

---

## Phase 3：业务逻辑叠加

基于 Phase 2 的 VL 导航能力，叠加具体业务：
- 养号浏览（自动浏览+点赞）
- 发布笔记（导航到发布页→输入内容→提交）
- 定时执行

## 技术细节

### VL Prompt 设计
```
分析这个APP截图，识别所有可交互元素。
返回JSON数组：
[{
  "label": "元素类型",
  "text": "元素上的文字",
  "x": 归一化x中心(0-1),
  "y": 归一化y中心(0-1),
  "w": 归一化宽度(0-1),
  "h": 归一化高度(0-1)
}]
```

### 节点点击
```javascript
// 手机端，归一化坐标 → 实际像素
var clickX = Math.round(node.x * device.width);
var clickY = Math.round(node.y * device.height);
click(clickX, clickY);
```

## 与 v1 的关键区别

| | v1 (YOLO) | v2 (VL驱动) |
|---|---|---|
| 识别方式 | 训练模型推理 | VL API 实时识别 |
| 部署依赖 | 需要训练环境 | 只需要 API |
| 精度 | 取决于训练数据 | VL 直接识别 |
| 延迟 | 低（本地推理） | 高（API调用30-60s） |
| 换手机 | 需要重新适配 | 归一化坐标，无需改动 |
| 成本 | 训练成本 | API 调用费用 |
